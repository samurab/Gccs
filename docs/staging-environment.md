# Staging Environment

Story 17.3 defines staging as the production-like release verification environment for the No-CUI MVP with synthetic CUI-ready demonstration workflows. Staging must deploy from CI/CD, use synthetic-only data, expose health checks for release smoke tests, and keep production customer data, production CUI, and production secrets out of the environment.

## Scope

- API: ASP.NET Core API published in Release configuration with `ASPNETCORE_ENVIRONMENT=Staging`.
- Web: React/Vite static build from `apps/web`.
- Database: dedicated staging PostgreSQL database, migrated from an idempotent EF Core script generated by CI.
- Object storage: dedicated staging bucket/container for synthetic/redacted test evidence and contract files.
- Cache: dedicated staging Redis-compatible cache.
- Queue and jobs: dedicated staging background job queue with job failure and backlog monitoring.
- Secrets: staging-only secret store values, separate from production credentials.

## Data Guardrails

- Staging is **No-CUI / compliance management only** with synthetic CUI-ready demonstration data.
- Staging uses synthetic-only pilot data.
- Staging contains no production customer data.
- Staging contains no real customer CUI.
- Staging must not reuse production secrets, production object storage buckets, production database snapshots, production unrestricted logs, or production customer uploads.
- Any restore into staging must be generated from sanitized fixtures or synthetic seed data.

## CI/CD Deployment

The GitHub Actions workflow `.github/workflows/staging.yml` runs on `workflow_dispatch` and pushes to `main` or `release/**`. It builds API and web artifacts, generates the EF Core idempotent migration script, validates the staging infrastructure contract, confirms the CUI-gating/no-production-data guardrails, performs the staging deployment step, and runs smoke tests.

The deployment environment must provide:

- `vars.STAGING_API_BASE_URL`
- `vars.STAGING_WEB_BASE_URL`
- Staging-only cloud identity, database, object storage, cache, queue, and secret-store access configured through the GitHub `staging` environment.

## Health And Smoke Coverage

Post-deployment smoke tests call `GET /health` on the staging API and require these signals:

- API status is `ok`.
- `dataPosture` is `No-CUI / compliance management only`.
- Database dependency reports `postgresql`.
- Cache dependency reports `redis`.
- Storage dependency reports `object-storage`.
- Background job dependency reports `background-jobs`.

The staging workflow uploads `staging-smoke-test-results` so success or failure is visible in CI/CD.

## Logs And Alerts

Staging observability must include:

- API request and error logs.
- Web deployment logs.
- Migration logs.
- Background job failure logs.
- Health degradation alerts.
- Queue backlog alerts.
- Job failure alerts.
- Upload/storage failure alerts.

Alerts route to the delivery/security owner for MVP hardening until a production on-call process is approved.
